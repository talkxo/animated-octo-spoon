Pluto CRM Developer Documentation

A guides, architectural charts, schemas, and standards reference for frontend developers.

System Overview

Pluto is a client-side **local-first CRM** application built with React, styled with vanilla CSS, and bundled using Vite. It offers dual execution modes: a fully functional local-offline mode utilizing `localStorage` caching and a real-time collaborative cloud mode integrated with Firebase (Authentication, Google Sign-In, and Firestore). In both configurations, the ultimate database source remains the user's private Google Spreadsheet, synced securely through a custom Google Apps Script Web App macro.

Vite + React App

Modular, functional architecture utilizing custom hooks to isolate auth actions, database operations, settings revisions, and local caches.

Local-First Engine

Operates offline first. Edits are committed immediately to the client-side database and buffered to an operation sync queue.

Google Sheets Sync

Synchronizes via Apps Script HTTP calls using base revisions and sequential flushing to prevent overwrites and resolve conflicts.

Dual Offline & Cloud Runtime Logic

To allow seamless local execution without requiring complex Firebase cloud hosting for every fork, the application implements a dynamic configuration detector. If Firebase API keys are missing in the environment, the app hides collaborative settings and runs inside an Offline Sandbox Mode.

Guard Enforcement Policy: Never display cloud-only visual containers when Firebase configuration is inactive. The isConfigured state is checked on load and supplied to SettingsView.jsx and Login.jsx to hide QR generation widgets and real-time team lists.

The Google Sheets Sync Engine

Writes to Google Sheets do not block user interactions. In auto mode, each edit triggers an optimistic write to the internal `leads` state, serializes the payload, schedules a job to append it to the client operation queue, and immediately attempts to flush the queue sequentially.

Sync Logic Breakdown & Edge Cases


1. **Payload Deduplication**: To avoid redundant roundtrips, the system checks the pending queue. If a lead update for lead-A is queued and the user modifies it again, the older payload is stripped, and the latest snapshot is placed at the end of the queue.

2. **Collision Guard (Revisions)**: Revisions prevent concurrent users from overwriting sheets data. Each lead record and settings block maintains a numeric `revision`. When posting an update, the current client revision is matched against the server's revision. If the server has a higher revision, a conflict is returned, the local queue is dropped, and a pull is forced to synchronize.

3. **Try/Finally Queue Locks**: The flush operation runs within a try/finally lock block using a ref marker (isSyncingRef.current). If an network/JSON parsing exception occurs inside the loop, the finally block releases the lock immediately, preventing system sync freezes.

4. **Opaque no-cors Fallback**: When users experience strict CORS policies (such as browser extensions or script setups), the system falls back to an optimistic no-cors HTTP POST. Because response bodies cannot be read in opaque mode, the queue pauses and schedules a reconciliation fetch 2.5 seconds later to read the revision back.

Google Sheets Database Schema

Google Sheets acts as the main backend. The Google Apps Script automatically bootstraps a workbook by creating the following sheets/tables if they do not exist:

Sheet Name Key Columns Data Format Description
Leads `id`, `name`, `company`, `phone`, `email`, `status`, `value`, `tags`, `pipelineId`, `revision` String & Numeric values Contains individual pipeline opportunities and version records for concurrent revisions.
Notes `id`, `leadId`, `text`, `timestamp`, `type` Log entry row strings Timeline interactions (calls, system prompts, manually entered logs, templates).
Settings `key`, `value` JSON configuration blobs Serialized pipelines, outreach slugs, preferences, and workspace variables.
CallingLists `id`, `name`, `leads`, `createdAt` Raw uploaded lead buffers Contains cold contacts imported via Excel/CSV queues for campaign sprinters.

UI & Typography Tokens System

Pluto CRM follows a high-density, flat styling language. We align UI scale boundaries according to standard modern SaaS scales (similar to Notion, Linear, and Figma) to provide visual consistency:

Token Name Rem Value Pixel Value Application Target
--text-title-1 `1.5rem` 24px Page headers and main analytics metric values.
--text-title-2 `1.125rem` 18px Modal titles, primary section headers, and glass card headers.
--text-title-3 `0.875rem` 14px KPI labels, card headers, secondary headings.
--text-body `0.85rem` 13.6px Standard descriptive text, checklist lines, dialog bodies.
--text-sm `0.8125rem` 13px Input fields, selection dropdowns, form labels.
--text-xs `0.75rem` 12px Button labels, mini status badges, tags.
--text-2xs `0.6875rem` 11px Timestamps, small meta-logs, sub-labels.
Visual Card Constraints: Cards (such as .glass-card and login page forms) must remain completely flat. Avoid incorporating box-shadow effects in CSS, as it breaks visual alignment with the design system.

Grid & Card Spacing Audit Rules

To maintain visual balance, never combine inline hardcoded margins (like `marginTop` or `marginBottom`) with flexible layouts. The layout follows a nested gap system:

Spacings Matrix


- **Card Column Gaps**: Grid elements and stack panels inside `settings-layout-column` rely solely on the flex `gap: 1.25rem` to space out consecutive cards.

- **Internal Card Padding**: Card elements (pipelines, WhatsApp templates, members lists) must use a standardized `padding: 1rem` on their container wrapper.

- **Header to Card Body**: The space between card header descriptions/divs and the first list element must be handled purely by the flex `gap: 1.1rem` from `.settings-card-body`. Never apply inline `marginBottom` or `marginTop` to header titles.

Setup & Local Environment Config

Follow these steps to copy the environment variable template and run the project locally:

Terminal commands 
# 1. Copy the environment config file
cp .env.example .env

# 2. Install dependencies
npm install

# 3. Boot Vite Development Server
npm run dev

# 4. Compile Production Bundles
npm run build

Fill out your configuration keys in your newly generated .env file:

.env Configuration File
# Server fallbacks (Sandbox Mode)
VITE_ADMIN_USERNAME=admin
VITE_ADMIN_PASSWORD=pluto2026

# Firebase Settings (Phase 2)
VITE_FIREBASE_API_KEY=AIzaSyA...
VITE_FIREBASE_AUTH_DOMAIN=pluto-app.firebaseapp.com
VITE_FIREBASE_PROJECT_ID=pluto-app
VITE_FIREBASE_STORAGE_BUCKET=pluto-app.appspot.com
VITE_FIREBASE_MESSAGING_SENDER_ID=78239011
VITE_FIREBASE_APP_ID=1:78239011:web:cfb10292

Sheets Apps Script Setup Guide

To authorize your Google Sheet endpoint, paste the Apps Script project configuration from the repository into Google Apps Script.

Deployment Constraint: During Apps Script deployment configuration, you must select **Execute as: Me (your-account@gmail.com)** and **Who has access: Anyone**. Selecting any other accessibility flag will block the frontend from performing sync writes.
