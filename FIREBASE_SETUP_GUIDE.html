🔥 Firebase Setup Guide for Beginners

Connect Pluto CRM
to Firebase

This guide walks you through setting up Firebase — Google's free cloud backend — so that Pluto CRM can use Google Sign-In, real-time workspace sync, team sharing with invite links, and a cloud-backed sync queue for the hybrid Firebase + Google Sheets setup. No prior Firebase experience needed.

⏱ ~10 minutes ✅ Beginner friendly 💸 100% free tier
✨ What you unlock after this guide
  • ✓ Google Sign-In — one-click login for you and your team
  • ✓ Real-time workspace sync — your Google Sheet URL, currency, sync mode, and shared workspace metadata saved to the cloud
  • ✓ Team workspace sharing — invite teammates via QR code; they join your workspace instantly
  • ✓ Secure invite tokens — 5-minute single-use links instead of raw sheet URLs in QR codes
  • ✓ Persistent sync queue — pending sheet writes survive refreshes and user sessions in Firebase mode

5

Set Firestore Security Rules

These rules match Pluto CRM's current hybrid architecture: shared workspace documents live under /workspaces, while each signed-in user owns their own /users/{uid} doc and /syncQueue subcollection.

  • 1 In the Firestore panel, click the "Rules" tab at the top.
  • 2 Delete everything in the editor and paste in the rules below. These same rules are also checked into the repo as firestore.rules.
  • 3 Click "Publish".
  • 4 If you prefer, you can copy from the local repo files instead of this page. The Firebase config files now tracked in the project are firebase.json, firestore.rules, and firestore.indexes.json. 
rules_version = '2';
service cloud.firestore {
  match /databases/{database}/documents {

    // Helper: check if the requesting user is a member of a given workspace
    function isMember(workspaceId) {
      return request.auth != null && get(/databases/$(database)/documents/users/$(request.auth.uid)).data.workspaceId == workspaceId;
    }

    // Helper: check if the requesting user is the owner of a given workspace
    function isOwner(workspaceId) {
      return request.auth != null && get(/databases/$(database)/documents/workspaces/$(workspaceId)).data.createdBy == request.auth.uid;
    }

    match /workspaces/{workspaceId} {
      // Only members of this workspace can read the workspace doc
      allow read: if request.auth != null && isMember(workspaceId);
      
      // Allow workspace creation if requesting user is setting themselves as creator
      allow create: if request.auth != null && request.resource.data.createdBy == request.auth.uid;
      
      // Only existing members can update the workspace doc
      allow update: if request.auth != null && isMember(workspaceId);
      
      // Only owners can delete the workspace doc
      allow delete: if request.auth != null && isOwner(workspaceId);

      match /members/{memberId} {
        // Any member can read the member list
        allow read: if request.auth != null && isMember(workspaceId);
        // Only owners can write member docs (except: a user joining via invite writes their own doc)
        allow write: if request.auth != null && (
          isOwner(workspaceId) ||
          request.auth.uid == memberId
        );
      }

      match /invites/{inviteId} {
        // Any member can create an invite
        allow create: if request.auth != null && isMember(workspaceId);
        // Anyone authenticated can read an invite (needed to validate it during join)
        allow read: if request.auth != null;
        // Anyone authenticated can update an invite (to mark usedBy during join)
        allow update: if request.auth != null;
        // Only owners can delete invites
        allow delete: if request.auth != null && isOwner(workspaceId);
      }
    }

    // Users — each user can only access their own document
    match /users/{userId} {
      allow read, write: if request.auth != null
                         && request.auth.uid == userId;

      match /syncQueue/{docId} {
        allow read, write: if request.auth != null
                           && request.auth.uid == userId;
      }
    }

  }
}
💡
What these rules mean: Workspace data (settings, members, invites) is only readable and writable by users who are actual members of that workspace — not just any logged-in user. Invite documents have a looser read/update rule so that incoming teammates can validate and claim a token during the join flow. Each user's personal /users/{uid} document and /syncQueue remain private — only they can access them.
6

Fill in the .env File

Paste the Firebase config values from Step 2 into your local environment file. The rest of the Firebase wiring is already checked into the repo.

  • 1 Open the file charming-oppenheimer/.env in your text editor.
  • 2 You'll see placeholder blank values for VITE_FIREBASE_*. Replace each blank with the matching value from your Firebase config object (from Step 2, Step 3).
  • 3 Save the file. The Vite dev server will hot-reload automatically.
  • 4 You do not need to add extra Firebase code by hand. Pluto CRM already reads these values from src/firebase.js and uses the tracked config files firebase.json, firestore.rules, and firestore.indexes.json.
🧩
What Firebase handles vs what Google Sheets handles: Firebase is the control plane for auth, workspace membership, shared settings, invite links, and queued sync metadata. Your actual CRM business data still lives in Google Sheets via the Apps Script backend.

Here's what your completed .env file should look like:

.env — fill in your values

VITE_ADMIN_PASSWORD=pluto2026

# Firebase — paste values from your Firebase console
VITE_FIREBASE_API_KEY=AIzaSyAbCd...
VITE_FIREBASE_AUTH_DOMAIN=pluto-crm-a1b2c.firebaseapp.com
VITE_FIREBASE_PROJECT_ID=pluto-crm-a1b2c
VITE_FIREBASE_STORAGE_BUCKET=pluto-crm-a1b2c.appspot.com
VITE_FIREBASE_MESSAGING_SENDER_ID=123456789012
VITE_FIREBASE_APP_ID=1:123456789012:web:abcdef
🔒
Never commit .env to Git. The .gitignore file already excludes it. If you accidentally push it, rotate your Firebase keys immediately in Project Settings → Service accounts.

Test the Full Firebase Flow

Start the dev server and verify Google Sign-In, workspace bootstrap, and Firestore-backed queue behavior all work end-to-end.

  • 1 If the dev server is already running, it will reload automatically when you saved .env. Otherwise run: npm run dev
  • 2 Open http://localhost:3000.
  • 3 The yellow "Firebase Configuration Notice" banner should now be gone, replaced by a "Continue with Google" button.
  • 4 Click the button. A Google sign-in popup will appear. Select your Google account.
  • 5 After signing in, you'll be taken to the main Pluto CRM dashboard. Your workspace is now saved to Firestore under /workspaces and your personal settings under /users/{uid}.
  • 6 Go to Settings. If you connect a Google Sheet URL, it should persist across refreshes because the workspace document now owns that setting.
  • 7 Optional but recommended: switch sync mode, theme, or currency and refresh the app. Those values should rehydrate from Firestore automatically after login.
  • 8 Teammate invite test (recommended): In Settings, find the "Workspace Invite Link" card (separate from the Sync to Mobile QR). Click Copy Link. Open it in a private/incognito browser window or a different browser. Sign in with a different Google account. Confirm: (1) you land in the owner's workspace — not a new empty one, (2) your email appears in the owner's Settings → Team Members panel, (3) the sheet URL is already populated without manual entry.

Troubleshooting Common Issues

Something not working? Check the most common problems below.

⚠️ Error: "auth/popup-blocked"

Your browser blocked the sign-in popup. Look for a blocked popup icon in the address bar and click it to allow popups from localhost:3000. Then try again.

⚠️ Error: "auth/unauthorized-domain"

Go to Firebase Console → Authentication → Settings → Authorized domains. Add localhost if it's not already there.

⚠️ Yellow banner is still showing after filling .env

The .env file must be saved and the Vite server must reload. Try stopping the server (Ctrl+C) and running npm run dev again. Also double-check that your VITE_FIREBASE_API_KEY is not empty or still says "your_api_key_here".

⚠️ "FirebaseError: Missing or insufficient permissions"

Your Firestore security rules haven't been updated, or you're not signed in. Make sure you pasted and published the rules from Step 5 so both /workspaces and /users/{uid}/syncQueue are allowed as expected. Also confirm you're signed in via Google before performing actions.

⚠️ Popup shows but signs in immediately without prompting

You might already be signed into only one Google account in your browser. The popup auto-selects it. This is normal behavior — you're already authenticated.

⚠️ Teammate joined but landed in their own empty workspace instead of mine

This was a known race condition that has been fixed. The old code would auto-create a workspace for any new user before checking for a pending invite. The fix: useWorkspace.bootstrap() now checks localStorage for a crm_pending_invite token before creating a new workspace, and skips creation if one is found. If you see this on an older deployment, redeploy with the latest code.

Also make sure you're using the "Workspace Invite Link" card in Settings (not the Sync to Mobile QR). The QR code is for personal device sync only — it does not add a teammate to your workspace.

⚠️ "FirebaseError: Missing or insufficient permissions" after updating rules

The new member-scoped rules require a user to already exist in /workspaces/{id}/members before they can read/write workspace data. If you pasted the rules manually in the Firebase Console, make sure you published them (not just saved as draft). Also confirm you deployed with ./node_modules/.bin/firebase deploy --only firestore:rules.

Skipping Firebase (Offline Sandbox Mode)

If you don't need team features, you can use Pluto CRM entirely offline with a local username and password.

ℹ️ How offline mode works

When the VITE_FIREBASE_* values in your .env are blank, Pluto CRM automatically enters local sandbox mode. Everything is stored in your browser's localStorage — no network requests, no account required.

Default login credentials (sandbox mode):

Sandbox credentials
Username: admin
Password: pluto2026

You can change these in your .env file by editing VITE_ADMIN_USERNAME and VITE_ADMIN_PASSWORD.
