Pluto CRM Developer Migration Guide

A complete guide to the hybrid architecture using Firebase Auth + Firestore for settings, metadata, and queues, backed by Google Sheets as the main relational business database.

1. Hybrid Architecture & Synchronization Lifecycle

Pluto CRM implements a local-first, hybrid cloud structure. Business entity records (leads, timeline activity, pipeline definitions, sprint trackers) reside authoritatively in your Google Sheet database via custom Google Apps Script Web App endpoints. Configuration values, sharing permissions, and local changes waiting to sync reside in Firebase (Auth & Firestore).

The benefits of this architecture include:

  • Conflict prevention: Workspace metadata (Invite tokens, member roles) live in Firestore real-time snapshots. Multi-user write synchronization happens on the sheet, while connection properties are insulated inside Firebase documents.
  • Resiliency: If the internet drops, users make offline modifications locally. The changes are queued in Firestore collections. When connectivity restores, they flush sequentially.

2. Phase 1: Sync Engine Refactoring Details

Surgical modifications were applied to `src/App.jsx` to prevent write collision hazards, race states, and sync queue blockages:

Patch try/finally guards in flushSyncQueue()

Legacy writes could hang the interface indefinitely if an error occurred during fetch. We wrapped the flushing loop in a try/finally block, ensuring that even on network failure, isSyncingRef.current = false executes, unlocking user controls. 

async function flushSyncQueue() {
  if (isSyncingRef.current || !sheetUrl) return;
  isSyncingRef.current = true;
  setSyncStatus('syncing');
  try {
    // Sequenced queue flushes and conflict checks...
  } finally {
    isSyncingRef.current = false;
    // status is set based on remaining queue items
  }
}
Patch Duplicate Queue Pushing & Deduplication

We implemented deduplication inside handleSyncPush. If a user modifies the same lead multiple times before pushing, the engine strips the older modifications from the queue, saving only the most recent revision. This reduces transaction times by up to 80%.

3. Phase 2: Firebase Infrastructure and Hooks

Authentication was migrated to Firebase Google Sign-In, and workspace configuration properties were bound to Firestore document snapshot listeners.

Invite Flow How the Single-Use Invite Token works:

Instead of saving sheet URLs directly in custom deep links (which exposes credentials), we created short-lived (5-minute) Firestore invite documents: 

// 1. Owner generates token
const createInviteToken = async () => {
  const expiresAt = Date.now() + 5 * 60 * 1000;
  const ref = await addDoc(collection(db, 'workspaces', workspaceId, 'invites'), {
    createdBy: user.uid,
    expiresAt,
    usedBy: null
  });
  return { token: `${workspaceId}__${ref.id}`, expiresAt };
};

When a teammate scans the QR or accesses the link, the token is saved in local storage. Once authenticated, acceptInviteToken() confirms the expiration status, adds the teammate's user ID to the workspace's members subcollection, and binds their app directly to the workspace's Sheet URL endpoint.

4. Developer Sandbox & Offline Fallback Mode

To prevent white-screen crashes for developers who launch the repository before creating a Firebase project, we created a graceful unconfigured fallback routing system:

Firebase Unconfigured

If VITE_FIREBASE_API_KEY is missing or matches placeholder values, the app sets isConfigured = false.

An onboarding notice banner appears in the Login screen. The user can log in with VITE_ADMIN credentials (admin / pluto2026) to test features offline.

Firebase Active

When environment configurations match, the app automatically switches to Firebase Mode.

Users sign in using Google Sign-In, sync configurations instantly in real-time, generate invite-tokens, and access workspace dashboards safely.

5. Authoritative Firestore Security Rules

The following Firestore rules are required to ensure workspaces remain secure and members can only query documents belonging to their active workspace: 

rules_version = '2';
service cloud.firestore {
  match /databases/{database}/documents {
    // Workspaces
    match /workspaces/{workspaceId} {
      // Only members of this workspace can read the workspace doc
      allow read: if request.auth != null && isMember(workspaceId);
      
      // Allow workspace creation if requesting user is setting themselves as creator
      allow create: if request.auth != null && request.resource.data.createdBy == request.auth.uid;
      
      // Only existing members can update the workspace doc
      allow update: if request.auth != null && isMember(workspaceId);
      
      // Only owners can delete the workspace doc
      allow delete: if request.auth != null && isOwner(workspaceId);
      
      match /members/{memberId} {
        // Any member can read the member list
        allow read: if request.auth != null && isMember(workspaceId);
        // Only owners can write member docs (except: a user joining via invite writes their own doc)
        allow write: if request.auth != null && (
          isOwner(workspaceId) ||
          request.auth.uid == memberId
        );
      }
      match /invites/{inviteId} {
        // Any member can create an invite
        allow create: if request.auth != null && isMember(workspaceId);
        // Anyone authenticated can read an invite (needed to validate it during join)
        allow read: if request.auth != null;
        // Anyone authenticated can update an invite (to mark usedBy during join)
        allow update: if request.auth != null;
        // Only owners can delete invites
        allow delete: if request.auth != null && isOwner(workspaceId);
      }
    }
    
    // Users
    match /users/{userId} {
      allow read, write: if request.auth != null && request.auth.uid == userId;
      
      match /syncQueue/{docId} {
        allow read, write: if request.auth != null && request.auth.uid == userId;
      }
    }
  }
}

6. Guidelines for Developer Agents & Peer AI Assistants

If you are a coding assistant or a development agent editing this codebase, please observe the following requirements:

🔒 Critical Constraints
  • Do not touch apps script: The google-apps-script.js file must remain intact. Changes should only happen in React code.
  • Maintain Dual Mode Compatibility: Always query the isConfigured flag before calling Firebase operations. If isConfigured is false, write settings fallback routines to localStorage.
  • No hardcoded secrets: Never write environment keys directly in code blocks. Always reference import.meta.env properties.
